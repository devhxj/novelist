# 章节创作工作流改进方案

## 一、当前问题

1. **单次生成**：整章由一次 LLM 调用完成，没有边写边思考的过程，质量不稳定
2. **上下文管理粗糙**：按层截断到固定字数，不是按相关性选择性注入
3. **缺少大纲环节**：直接从上下文跳到完整章节，没有结构规划
4. **模式选择冗余**：Agent/Plan/Review 三种模式与子 Agent 功能重叠，增加复杂度
5. **LLM 自主收集上下文不可靠**：依赖 LLM 自己调 MCP 工具查信息，容易遗漏

---

## 二、上下文注入架构

**对话开始时就默认注入核心上下文，不等用户开口。**

用户打开创作工作台 → 系统自动注入：

```
[system] 系统指令 + 创作偏好 + 写作规则（固定）

[system 消息] 自动注入：
  · 故事状态文档（角色当前状态、活跃伏笔、剧情线进展）
  · 角色索引表（所有角色一行概要）
  · 最近 2-3 章全文
  · 下一章待办（时间线中 pending 的任务、要回收的伏笔、要推进的节点）
```

好处：
- AI 一上来就知道"故事现在在哪、接下来该做什么"
- 用户说"继续写"时不用再等 AI 收集上下文
- 即使用户只是讨论剧情，这些上下文也有帮助
- 这些内容在稳定前缀里，缓存命中率高

---

## 三、章节创作流程

```
用户："帮我写第15章"
    ↓
【已有上下文】故事状态 + 角色 + 最近章节 + 下一章待办（已注入）
    ↓
【生成大纲】LLM 基于已有上下文直接生成结构化大纲
  · 场景节拍（分几个场景，每个场景发生什么）
  · 涉及角色、情绪走向、关键对话
  · 伏笔操作（埋设/回收点）
  · 章末钩子
    ↓
【用户审批】可要求修改大纲
    ↓
【精确补注入】根据大纲内容追加到 messages：
  · 特定角色的完整档案 + 记忆检索
  · 涉及地点的详细设定
  · 伏笔原文片段
    ↓
【写作】LLM 基于完整上下文 + 大纲写正文
    ↓
【写后自动执行】
  · 更新故事状态文档
  · 更新时间线（标记已完成的待办）
  · 生成章节摘要
  · 向量记忆入库
```

---

## 四、其他意图的处理

| 用户意图 | 处理方式 |
|---|---|
| 创建新章节 | 走上述大纲流程 |
| 修改/扩写/缩写 | LLM 直接调 `edit_chapter`，已有上下文足够 |
| 讨论剧情/角色 | LLM 直接对话，按需调工具查详细信息 |
| 查看信息 | LLM 调查询工具 |

不需要用户选模式，系统根据意图自动判断。

---

## 五、上下文分层（按缓存稳定性排列）

```
=== 稳定前缀（高缓存命中）===
[system]     系统指令 + 创作偏好 + 写作规则          ← 完全不变
[msg1]       故事状态文档                            ← 每章微调 5-10%
[msg2]       角色索引表                              ← 很少变
[msg3]       最近 2-3 章全文                         ← 小窗口滑动
[msg4]       下一章待办（时间线 pending 项）          ← 每章变化

=== 每章变化（放后面）===
[user]       用户请求
[system]     精确补注入（大纲确定后的角色记忆等）
[assistant]  大纲
[user]       用户审批
[assistant]  正文
```

弹性层根据模型窗口自适应分配（64K/200K/1M）。

## 六、故事状态文档（核心创新）

不是时间线（面向未来的待办清单），而是故事当前存档（面向现在的状态快照）。
每次写完章节后低成本 LLM 调用自动更新，把几百万字压缩成 ~5K tokens 的实时快照。

结构化格式示例：

```
## 角色状态
- 林风：炼气期三层，青云宗外门，与苏婉关系暧昧，不知道苏婉真实身份
- 苏婉：天机阁圣女伪装中，对林风产生真感情

## 活跃伏笔
- [发展中] 林风体内印记 → 第8章出现，未解释
- [即将回收] 苏婉真实身份 → 已铺垫怀疑线索，预计2-3章内揭露

## 剧情线进展
- 主线·修仙之路：林风通过外门考核，即将进入内门
- 感情线·林苏：互有好感未表白，隐瞒身份是障碍

## 已揭示世界观
- 读者已知：天机阁在暗中观察林风（第12章视角切换）
- 角色已知：林风不知道天机阁存在
```

**与时间线的区别：**
- 时间线 = 待办清单，面向未来（"接下来要做什么"）
- 故事状态文档 = 存档快照，面向现在（"故事现在是什么情况"）

---

## 七、子 Agent 调整

- **去掉** Writer Agent（主 LLM 在完整上下文下直接写）
- **去掉** Agent/Plan/Review 三种手动模式（改为意图自动识别）
- **保留** Review Agent（写后质量审查）
- **保留** Memory Agent（记忆检索）

---

## 八、缓存策略

- 稳定内容放前面，变化内容放后面
- 最近章节用小窗口（2-3章），靠故事状态文档覆盖历史信息
- 连续创作场景下自然命中，不刻意追求高命中率
- 200K 窗口下不做激进压缩，先跑通再优化
